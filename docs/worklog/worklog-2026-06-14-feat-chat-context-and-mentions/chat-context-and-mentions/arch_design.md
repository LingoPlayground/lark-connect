# 群上下文与成员查询架构说明

## 背景

早期 daemon 设计把 `src/daemon/runtime.js` 定义为本地状态机：负责绑定、消息队列、去重、等待者、确认状态和空闲时间。`src/daemon/http-server.js` 负责本机 HTTP 协议边界，把 MCP 或 CLI 请求映射到 runtime。

本功能新增的聊天上下文、群成员和群内机器人查询不是本地状态机行为。它们是对当前绑定聊天的按需外部读取，不进入待处理消息队列，也不需要确认。

## 决策

保持 runtime 只管理本地内存状态；HTTP server 在校验本地绑定后，把绑定得到的 `chatId` 传给对应 Lark 客户端完成外部读取或发送。

因此：

- `runtime` 继续拥有绑定、队列、去重、等待、确认和资源可见性不变量。
- `http-server` 拥有本机协议解析、输入校验、绑定解析、错误码映射，以及把已绑定 `chatId` 交给 Lark 客户端的编排职责。
- `src/lark/*` 客户端拥有飞书接口细节、分页参数、超时和返回值规整。
- MCP 层继续只调用本地 daemon HTTP client，不直接依赖飞书 SDK。

## 取舍

这个决策让上下文读取和成员查询避免进入 runtime，从而不增加新的持久化或消息状态实体，也避免历史消息被误认为需要 `ack`。

代价是 HTTP server 不再是只调用 runtime 的纯薄适配层；它承担了“已绑定聊天上的外部动作编排”。当前接受这个代价，因为发送文本、文件、图片、视频、下载资源和新增查询都需要同样的绑定解析步骤。若后续这类路由继续增长，应考虑把已绑定聊天动作抽到独立的 daemon service 模块，避免 HTTP server 继续膨胀。

## 不变量

- 历史上下文读取不进入待处理队列。
- 历史上下文读取不需要 `lark_connect_ack_message`。
- 群成员和群内机器人查询不改变绑定状态。
- 所有外部飞书调用必须有有界超时，并把失败映射成结构化 daemon 错误。
