# Search chat tool — Spec (聊天搜索工具 — 规范)

> 让智能体能自行搜索机器人可见群聊，减少人工获取 `chatId` 的成本，并让已知 `chatId` 的单聊也能绑定和接收消息。

## Why (为什么做)

当前群消息响应技能要求用户先提供 `chatId`，但普通用户很难直接拿到飞书群聊 ID。更自然的流程是用户告诉智能体群名，智能体通过 MCP 工具搜索机器人可见的群聊，找到后直接绑定；如果找不到，再提示用户创建群或把机器人拉进目标群。

同时，飞书事件里单聊也有 `chatId`。如果用户已经有一个机器人单聊会话，系统不应该在本地路由层天然排斥它；只要明确绑定到该 `chatId`，智能体就可以把单聊当作一个会话目标处理。

## Findings (调研发现)

- [README.md](../../../README.md) 当前安装和运行说明要求绑定时传入 `chatId`，但没有提供获取群聊 ID 的工具。
- 变更前的主响应技能要求用户提供 `chatId`，并且技能名偏“群消息响应”，不适合同时覆盖配置、搜索、绑定和单聊。
- 飞书官方文档 `im.v1.chat.search` 是 `GET /open-apis/im/v1/chats/search`，用于搜索用户或机器人可见的群列表；官方描述强调它搜索群聊，不是私聊发现接口。
- `@larksuiteoapi/node-sdk` 已暴露 `client.im.v1.chat.search`，注释同样说明它搜索对用户或机器人可见的群列表。
- `@larksuiteoapi/node-sdk` 的消息策略对群聊和单聊分开处理：群聊可要求提及机器人，单聊由 `dmMode` 控制。当前仓库把 `dmMode` 设置为 `disabled`，且 daemon runtime 也要求 `mentionedBot`，所以单聊消息目前不会进入队列。
- 本仓库没有 `docs/rules/spec/` 项目专属 spec 规则。

## What (做什么)

### 1. 聊天搜索工具

MCP 新增一个搜索聊天工具。智能体可以传入关键词，获得机器人可见的候选群聊列表。返回项至少包含群聊 ID、名称、描述、类型和成员数量等可用于判断的字段。

当搜索结果为空时，工具返回结构化空结果，并包含面向智能体的下一步提示：请用户确认群名；如果目标群不存在，则创建群；如果群存在但不可见，则把对应飞书应用的机器人拉入群并重试。

搜索工具不要求先绑定会话，但要求本地守护进程已启动并配置了飞书应用凭据。

### 2. 技能改名和使用流程

面向智能体的主要技能改名为 `lark-connect`。它覆盖配置检查、搜索群聊、绑定会话、发送资源、等待消息和确认消息。

技能流程调整为：如果用户没有提供 `chatId`，先用搜索工具按群名查找；只有搜索不到或结果不明确时，才向用户要更多信息或提醒用户创建群并拉入机器人。

### 3. 单聊绑定支持

对于已知 `chatId` 的单聊，绑定、轮询、等待、发送和确认流程应与群聊一致。

群聊仍然只接收明确提及机器人的消息。单聊不要求提及机器人，但仍必须先绑定到对应 `chatId` 才会被当前智能体会话领取。

## Out of scope (本次不做)

- 不做私聊发现或按用户名创建单聊，因为当前确认的飞书搜索接口只覆盖群聊。
- 不做多群或多会话同时绑定；当前仍保持一个绑定。
- 不新增持久化状态；搜索结果和绑定状态仍不写数据库。
- 不改变发送图片、视频、文件等工具的语义。
