# 群上下文与 @ 协作

## 背景

lark-connect 的目标是让本地个人 Agent 进入真实团队协作场景。Agent 不只是在被 mention 后响应单条消息，还需要能主动理解群聊里的近期上下文，并能通过回复和 @ 与人类同事、其他同事的 Agent 协作。

## 用户需求

1. Agent 绑定飞书群聊或单聊后，可以获取该聊天的近期消息，默认最近 10 条，用于开始工作前理解上下文。
2. Agent 发送文本消息时，可以回复某条原始消息。
3. Agent 发送文本消息时，可以 @ 人类用户或其他机器人；也可以只发送 @，不附加正文。
4. Agent 收到实时消息时，可以直接看到发送者类型；如果发送者是机器人或应用，消息 payload 不能丢失这个信息。
5. Agent 绑定群聊后，可以查询群成员和可识别到的群内机器人，便于选择 @ 对象和判断消息来源。
6. 技能文档需要告诉 Agent 什么时候先读取上下文、什么时候继续短等待或心跳监听。
7. README 需要补充项目背景：个人 Agent 进入真实团队，支持 Agent 与同事、Agent 与其他 Agent 交流。

## 非目标

- 不新增持久化实体。群上下文来自飞书消息历史接口，不保存在本地数据库。
- 不做联系人或机器人目录搜索。@ 目标由调用方传入飞书可识别的用户或机器人标识。
- 不改变现有“只响应 mention 消息”的监听规则。
- 不改变文件、图片、视频发送接口的回复能力。
- 不承诺完整枚举所有群内机器人。飞书群成员接口不会返回机器人成员，机器人列表只能来自当前应用身份和近期聊天上下文中可识别的应用发送者。

## 验收标准

1. 新增 MCP 工具可以对当前绑定聊天读取最近消息；未传数量时默认 10 条。
2. 上下文工具在 session 未绑定时返回现有风格的结构化错误，并提示先绑定。
3. 上下文结果包含足够协作信息：消息标识、聊天标识、发送者信息、消息类型、文本或内容摘要、创建时间、回复关系与 mention 信息。
4. `lark_connect_send_message` 支持 `replyToMessageId`，并继续返回回复相关字段。
5. `lark_connect_send_message` 支持结构化 `mentions`，每个 mention 可以表示人类或机器人；当只有 mention 没有正文时也能发送。
6. `lark_connect_wait_messages` 和 `lark_connect_poll_messages` 返回的实时消息 payload 包含发送者类型；当飞书原始事件给出 `sender_type=app` 时结果中可见。
7. 新增 MCP 工具可以查询当前绑定群聊的人类成员，并返回可识别到的机器人列表和覆盖范围说明。
8. 技能文档与 README 说明新能力的使用场景和最佳实践。
