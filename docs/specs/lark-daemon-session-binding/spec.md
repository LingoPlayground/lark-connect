# Lark Daemon Session Binding — Spec (飞书守护进程会话绑定 — 规范)

> 定义飞书长连接守护进程、Agent 会话绑定、消息领取与确认的第一版外部行为。

## Why (为什么做)

飞书群里的产品和设计反馈需要稳定进入本地 Codex 或 Claude Code 会话。一次性监听适合调试，但真实使用时，Agent 会话不应该自己持有飞书长连接；更合理的边界是由本地守护进程持续接收群消息，再由当前 Agent 会话通过 MCP 工具领取属于自己的反馈。

第一版要保持小而可用：状态只保存在守护进程内存里，不引入数据库、状态文件或后台自动拉起机制。用户明确启动守护进程，Agent 明确绑定群聊和会话，消息显式确认后才算处理完成。

## Findings (调研发现)

- [docs/research/lark-agent-bridge.md](docs/research/lark-agent-bridge.md) 记录了飞书真实群验证：`node-sdk Channel` 能收到目标群里的结构化机器人提及，并返回 `mentionedBot`、`mentions`、`resources` 等字段。
- [src/lark/listener.js](src/lark/listener.js) 已有一次性监听能力，当前按 `chatId` 和 `mentionedBot` 过滤消息，并归一化输出。
- [src/cli.js](src/cli.js) 当前已有 `listen --once`，但它是调试型命令，不提供多次消息领取、会话绑定或守护进程生命周期。
- [AGENTS.md](AGENTS.md) 要求如无必要勿增实体；本 spec 因此不引入持久化存储。
- 当前仓库没有 `docs/rules/spec/`，无项目专属规格规则。

## What (做什么)

### 1. 守护进程生命周期

CLI 必须提供一个显式启动本地守护进程的入口。守护进程启动后建立飞书长连接，并对本机 MCP/CLI 调用暴露本地接口。MCP 工具不得自动启动守护进程。

当守护进程不在线时，MCP 工具必须返回明确错误，包含机器可读错误码、面向 Agent 的说明，以及可执行的启动命令提示。错误码为 `DAEMON_NOT_RUNNING`。

守护进程在 1 小时内没有收到任何飞书事件，也没有收到任何本地 MCP/CLI 调用时，必须自动关闭。关闭后下一次 MCP 调用仍按 `DAEMON_NOT_RUNNING` 处理。

第一版守护进程状态只存在内存中。守护进程退出或重启后，绑定、待处理消息、已投递未确认消息和去重集合全部丢弃。

### 2. 会话绑定

MCP 必须提供绑定工具，把一个飞书群绑定到一个 Agent 会话。绑定参数必须包含：

- 飞书群 `chat_id`
- Agent 类型，例如 Codex 或 Claude Code
- Agent 会话标识，也就是 Codex thread 或 Claude Code session
- 工作区路径
- 是否替换已有绑定

第一版强约束：同一个飞书群同一时间只能有一个活跃绑定。若群已绑定且新请求没有显式替换，绑定必须失败并返回明确的冲突错误。若请求显式替换，旧绑定被移除，旧会话未确认消息被丢弃，新绑定成为该群唯一活跃绑定。

飞书消息线程字段可以作为消息元数据保留，但不参与第一版主路由。第一版主路由只按 `chat_id -> agent_session_id` 投递。

### 3. 消息接收与消费

守护进程收到飞书群里提及机器人的消息后，若该群存在活跃绑定，消息进入对应 Agent 会话的内存队列。若不存在绑定，消息可以被忽略或记录为无法路由，但不得投递给其他会话。

Agent 通过 MCP 工具领取消息。领取消息后，消息不得立刻删除；只有 Agent 调用确认工具后，消息才算已消费。第一版消息状态至少需要表达：

- 已接收
- 待领取
- 已投递但未确认
- 已确认

MCP 必须同时支持立即轮询和限时等待两种领取模式。立即轮询没有消息时返回空结果；限时等待在超时前收到消息则返回消息，否则返回空结果或超时状态，但不得视为错误。

### 4. 去重与空闲刷新

守护进程必须在内存中做短期事件去重，避免长连接重连或飞书重复投递造成同一条消息被重复交给 Agent。

以下活动必须刷新守护进程空闲时间：

- 收到任意飞书事件，即使该事件最终没有匹配绑定
- 收到任意本地 MCP/CLI 调用

### 5. 一次性监听命令定位

`listen --once` 保持为调试和烟雾测试工具。它可以继续直接建立飞书长连接并等待下一条目标群 mention，但不是第一版 Agent 消息消费主路径。正式 Agent 消费路径必须经过守护进程和 MCP 绑定。

## Out of scope (本次不做)

- 不做持久化存储；守护进程重启后不恢复绑定或未确认消息。
- 不补拉守护进程离线期间的飞书历史消息。
- 不支持同一个飞书群同时绑定多个活跃 Agent 会话。
- 不做基于飞书 `root_id` 或 `thread_id` 的多会话并发路由。
- 不实现截图、录屏、文件上传等出站产物发送能力。
- 不自动发现 Codex thread 或 Claude Code session 标识；第一版由 MCP 工具参数传入。
- 不要求 MCP 工具自动启动守护进程。

## Open questions (悬而未决)

1. 本地 HTTP 端口默认值归 plan / impl 阶段决定；需求层只要求默认监听本机回环地址，并允许后续配置覆盖。
2. 本地 HTTP 是否需要启动时 token 鉴权归 plan / impl 阶段决定；第一版需求层只要求不监听公网地址。
3. 无绑定消息是否需要在 daemon 日志中可见归 plan / impl 阶段决定；需求层只要求不得误投递给其他会话。
